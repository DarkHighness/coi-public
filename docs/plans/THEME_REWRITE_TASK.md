# `zh/themes.json` 重写任务规范

本规范的核心原则：**分类驱动，类型专精**。
不同类型的题材有不同的叙事规则和爽点节奏，不能用同一套写法。

---

## 1. 任务目标

对 `src/locales/zh/themes.json` 的每个主题进行深度重写，产出可作为后续剧情扩写地基的文本。

### 成功标准

1. **类型准确**：正确识别题材类型，并遵循对应规则
2. **题材可区分**：不同主题在叙事语气、冲突结构、世界运转逻辑上有明显差异
3. **世界可运行**：不是背景板；玩家不行动时，世界也会继续推进
4. **主角有生态位**：主角是系统中的一个节点，不是世界中心
5. **开场风格正确**：符合对应类型的开场要求
6. **文本去模板化**：避免 AI 腔、套话、复读句群
7. **反模板禁句清零**：下列句群在 `batch_0002~batch_0069` 出现次数必须为 0
   - `秩序重组期`
   - `旧规则还在执行`
   - `新规则已经试运行`
   - `至少三方并存`
   - `个人抉择放进更大的局势变化中`
   - `远景交代版图与阵线`
   - `近景落到人物呼吸、伤口和临场动作`

## 2. Prompt 体系结构（V3）

### 核心文件

```
docs/plans/
├── THEME_REWRITE_PROMPT_V2.md    ← 通用核心规则
├── THEME_MAPPING_TABLE.md        ← 题材分类对照表
├── PROMPT_SHUANGWEN.md           ← 爽文类专用规则
├── PROMPT_NUEWEN.md              ← 虐文类专用规则
├── PROMPT_ZHIYU.md               ← 治愈类专用规则
├── PROMPT_XUANYI.md              ← 悬疑恐怖类专用规则
└── PROMPT_SHISHI.md              ← 史诗世界观类专用规则
```

### 使用流程

1. **识别题材类型**：查阅 `THEME_MAPPING_TABLE.md`
2. **加载通用规则**：应用 `THEME_REWRITE_PROMPT_V2.md`
3. **加载类型规则**：根据类型加载对应的 `PROMPT_*.md`
4. **按规则生成**：遵循两份规则的要求生成内容
5. **验证检查**：确保符合类型要求和禁忌

## 3. 题材类型速查

| 代号 | 类型 | 核心特征 | 开场风格 |
|------|------|----------|----------|
| `SHUANGWEN` | 爽文类 | 身份反转、打脸、逆袭 | 冲突前置式 |
| `NUEWEN` | 虐文类 | 误会、牺牲、遗憾 | 余韵式 |
| `ZHIYU` | 治愈类 | 温暖、成长、救赎 | 氛围式 |
| `XUANYI` | 悬疑恐怖类 | 信息差、推理、恐惧 | 异常式 |
| `SHISHI` | 史诗世界观类 | 宏大叙事、种族/势力 | 命运召唤式 |
| `IP` | IP 改编类 | 基于特定作品 | 根据原作类型 |

---

## 4. 可改字段与边界

每个主题仅可改写：

- `name`
- `narrativeStyle`
- `backgroundTemplate`
- `example`
- `worldSetting`
- `protagonist_preference`

禁止：

- 修改主题 key
- 新增/删除主题
- 修改非上述字段
- 输出占位符（TODO、待补、示例占位）

---

## 5. 各类型开场风格要求

### 爽文类（冲突前置式）

- 第一句必须包含冲突/反差
- 禁止环境描写开头
- 结尾给出**下一步行动点**或**未决矛盾**（不写“下一周目/重玩”式 meta 口吻）
- 尽量有一句可传播的“名场面短台词”（不强制每条都硬塞）

### 虐文类（余韵式）

- 从已发生的悲剧痕迹开始
- 情绪是隐痛，不直接写悲
- 结尾是隐性绝望

### 治愈类（氛围式）

- 从温暖的细节开始
- 情绪是微甜回忆
- 结尾是期待

### 悬疑恐怖类（异常式）

- 从微小的不对劲开始
- 信息差设计明确
- 结尾是深渊凝视

### 史诗世界观类（命运召唤式）

- 平静被打破的瞬间
- 视觉分镜感
- 结尾是使命开启

---

## 6. 输出格式

只输出：

```json
{
  "<theme_key>": {
    "name": "...",
    "narrativeStyle": "...",
    "backgroundTemplate": "...",
    "example": "...",
    "worldSetting": "...",
    "protagonist_preference": ["..."]
  }
}
```

不输出解释文本，不输出 markdown 包裹，不输出额外字段。将你的输出放置到 output.json 文件中。

## 7. 验证检查清单

### 类型一致性检查

- [ ] 已正确识别题材类型
- [ ] `example` 开场风格符合类型要求
- [ ] 包含该类型的必备元素
- [ ] 未触犯该类型的禁忌

### 通用质量检查

- [ ] 题材没有串味，世界规则自洽
- [ ] 主角不是世界中心
- [ ] 第一决策明确，代价具体
- [ ] 文风自然，无模板痕迹
- [ ] `example` 有名场面台词

### 世界底座检查（必须落地成“制度/流程/器物”，不要喊口号）

- [ ] `worldSetting` 明确 **时间 / 地点 / 人物-势力**（小说三要素）
- [ ] 至少 1 个“可操作机制”（机构如何运转、资源如何分配、流程如何审批/交易/惩罚）
- [ ] 至少 2 个可触摸的场景名词（地点/器物/手续/记录/符号/票据）
- [ ] 至少 1 条“世界自推进机制”（倒计时/竞争者/恶化链/错过窗口；写清“不行动会发生什么”）
- [ ] 至少 2 条“非主角行动线”（别人在推进什么？制度在推什么？对手在争什么？）

### IP 题材额外检查

- [ ] 保留了原作硬设定
- [ ] 未跨 IP 拼接
- [ ] 能让粉丝产生代入感
- [ ] 若不熟悉该 IP：**只扩写原字段中已明确的世界规则/术语**，避免新增会硬冲突的事实（必要时列入“待人工复核清单”，不要在 `output.json` 里写备注）
